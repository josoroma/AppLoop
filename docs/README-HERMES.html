AppLoop · Local runtime assets

.apploop structure & .hermes agent surface

AppLoop keeps two root-local trees with different jobs. .apploop/ is the product’s mutable runtime state (SQLite, generated workspaces, logs). .hermes/ is the repo-local Hermes home: agent markdown, the /ui-builder bundle, skills, hooks, commands, plus gateway operational caches. The builder UI never talks to Hermes by itself. It posts to AppLoop (/api/chat or server actions); AppLoop composes an agentBundle from those Hermes assets and forwards it through the local gateway.

01 · Contracts

Two roots, two contracts

Mutable.apploop/

Product-owned runtime. SQLite inventory, generated Next.js project workspaces, per-project git checkpoints, and runtime process logs. Created/wiped by make seed / make apploop-reset.

Declarative.hermes/agents|bundles|skills|hooks|commands

Versioned agent operating system for AppLoop project runs. Loaded into the agentBundle AppLoop ships on every Hermes run so the gateway does not improvise process from empty context.

Operational.hermes caches/state

Gateway side effects: sessions dumps, model caches, logs, auth files, desktop stamps. Local-only kit that must not be treated as product schema.

02 · Product state

./.apploop structure

Default layout after seed/dev. Paths come from env defaults (PROJECTS_ROOT=.apploop/projects and DATABASE_URL=file:.apploop/builder.sqlite.

  • .apploop/ — local AppLoop runtime root (gitignored)
    • builder.sqlite — Drizzle SQLite: projects, conversations, messages, runs, runtimes, themes, settings, templates, checkpoints…
    • projects/ — one generated Next.js app per project slug
      • / e.g. ai-engineer-cv, luma-admin, default-next-js
        • app/ route modules + globals.css (theme applied here)
        • components/ includes inspector-provider + theme-provider
        • package.json / node_modules/
        • .next/ preview build cache
        • .git/ per-project git used by createFileSnapshot / restore
    • runtime-logs/
      • .log stdout/stderr from preview next dev
    • optional: *.db backups, .trash/ for deleted workspaces

Who writes where

PathWriter
builder.sqlite AppLoop repository layer (lib/projects/repository.ts) via actions + /api/chat
projects//* create/duplicate actions locally; Hermes tools only when run workspacePath points here
projects//.git AppLoop createFileSnapshot / revertToFileSnapshot
runtime-logs/ AppLoop runtime supervisor capturing preview process output

Template-edit exception

Template authoring/editing binds a project row whose workspacePath points at templates/ (repo tree), not .apploop/projects/…. SQLite still lives under .apploop/; the durable template files live under templates/. 

.apploop/builder.sqlite
  projects.workspace_path = /…/AppLoop/templates/vestaboard
  projects.name           = "Template: Vestaboard"

# vs normal project
  projects.workspace_path = /…/AppLoop/.apploop/projects/ai-engineer-cv
Mental model: .apploop is the running product database + customer workspaces. Hermes is invited into a single workspace path under containment rules. Sibling projects and the builder source tree are out of scope for project-edit runs.

05 · Bundles

.hermes/bundles/

Currently one product bundle: ui-builder at .hermes/bundles/ui-builder/BUNDLE.md, exposed as slash command /ui-builder.

Activation order (from BUNDLE.md)

  1. /security-review — isolation first
  2. /hermes-gateway — server-only stream contract
  3. /visual-selector — narrow inspect targets
  4. /theme-system — resolve Luma tokens
  5. /frontend-design — structure UI
  6. /generated-app-standards — code conventions
  7. /project-runtime — preview/runtime checks

Role in gateway payload

agentBundle.skillBundle = {
  id: "ui-builder",
  command: "/ui-builder",
  path: ".hermes/bundles/ui-builder/BUNDLE.md",
  skills: [ ...ordered skill defs ],
  activationOrder: [...]
}

The bundle is the glue metadata telling Hermes which skills load and in what order, not a separate runtime process.

The directory holds many general Hermes skills. AppLoop’s project UI builder pathway only registers seven of them into UI_BUILDER_SKILL_BUNDLE (lib/hermes/skills.ts).

Other skills in .hermes/skills/* (research, creative, mlops, apple, …) exist for Hermes desktop/chat profiles. They are not auto-forwarded by AppLoop’s createProjectAgentBundle. The builder pathway is intentionally the ui-builder seven.

10 · Wire path

End-to-end communication sequence

1. Browser (BuilderShell)
     useChat({ id: projectId }) → POST /api/chat { id, messages }

2. AppLoop app/api/chat/route.ts
     requireProjectAccess → load overview from .apploop/builder.sqlite
     detect mode via getTemplateIdFromWorkspacePath(workspacePath)
     INSERT messages(user) + runs(running)
     agentBundle = createProjectAgentBundle({
       projectId, workspacePath, theme/settings,
       mode: "project-edit" | "template-edit"
     })
       // bundle includes:
       //   orchestrator + delegates paths under .hermes/agents/*
       //   skillBundle → .hermes/bundles/ui-builder + 7 skills
       //   hooks → 4 HOOK.md paths
       //   commands → 7 command md paths
       //   isolationRules + completionCriteria

3. HermesClient.streamProjectRun / runProjectOnce
     HTTP to local gateway (HERMES_HOME=.hermes)
     payload carries agentBundle + workspacePath + composed message (+ images)

4. Gateway (Hermes process)
     Uses HERMES_HOME assets; follows /ui-builder activation order
     Tools mutate ONLY workspacePath (.apploop/projects/ or templates/)
     Emits session / run / activity / text-delta / done events

5. AppLoop maps stream → AI SDK UI chunks
     INSERT assistant message; UPDATE runs terminal status
     Browser shows Hermes bubble + activity cards
     Preview iframe reflects file changes via HMR/reload

11 · Modes

Run modes

modeHow chosenWritable rootIsolation highlight
project-edit /api/chat and workspace not under templates/ .apploop/projects/ Never touch AppLoop source, templates/, or sibling projects
template-edit Chat on a project whose workspacePath is a templates/ id templates/ Keep template- prefix + standalone package/app/layout providers
template-authoring Create Template action’s runProjectOnce new templates/ Same as template-edit; first generative pass before redirect

Bundle contents (agents/skills/hooks/commands) are the same object shape; only projectContext.mode + isolationRules + synthetic projectId differ.

12 · Hygiene

What not to commit

Leave local

  • All of .apploop/
  • .hermes/sessions/, logs/, cache/, state/
  • Model / provider caches, auth.json, gateway state stamps
  • Runtime node_modules / .next inside seeded projects

Do commit (product contracts)

  • .hermes/agents/*.md
  • .hermes/bundles/ui-builder/BUNDLE.md
  • AppLoop skill set under .hermes/skills/{frontend-design,generated-app-standards,…}
  • .hermes/hooks/*/HOOK.md
  • .hermes/commands/*.md
  • lib/hermes/{agents,skills,hooks,commands,client}.ts that wire them
Debugging tip: if chat ignores inspect targets, verify the composed prompt contains preferredSelector JSON and that gateway runs still include skillBundle.path = .hermes/bundles/ui-builder/BUNDLE.md with visual-selector in activation order. If writes escape the workspace, reconcile isolationRules in createProjectAgentBundle and the project-scope-guard hook.
