AppLoop · Architecture

UI, Database, AI Gateway & API Communication

AppLoop is a local-first visual builder for generated Next.js apps. The browser owns interaction and framing. The Next.js process owns project state, security-sensitive paths, and runtime control. Hermes owns generative workspace edits through a local AI gateway. This document is the engineer-facing map of how those planes connect in production-shaped operation — not marketing copy.

Builder :3001 Preview apps :3100–3199 Hermes gateway default :8642 State SQLite + workspace FS + git Theme shadcn/Luma dark tokens
Control plane
AppLoop Next.js

Routes, server actions, SQLite ownership, preview process lifecycle, authz for project ids.

Compute plane
Hermes gateway

Streaming agent runs bound to a workspace path and AppLoop-provided agent bundle.

Presentation plane
Builder + preview iframe

Chat/inspect UX in the parent; generated app runs isolated on a reserved preview port.

01 · Topology

System topology

Three processes cooperate on one machine. There is no multi-tenant cloud control plane. All sensitive keys stay server-side.

02 · Contracts

Ownership boundaries

The clean split prevents the model runtime from becoming a general OS agent and prevents the browser from becoming an authoritative state store.

OwnsAppLoop

Project/template inventory, SQLite truth, preview ports/PIDs, path containment, token redact surfaces, theme validation, redirect policies, agent-bundle assembly, and conversation durability.

OwnsHermes

Model inference orchestration, tool loop for file edits/validation inside the provided workspace, streaming event semantics, session continuity for generative runs.

OwnsBrowser

Interaction state (inspect selection, composer UI, panel layout), SSE consumption for chat, preview framing, and optimistic transcript updates before durable round-trips finish.

NeverCrosses

Browser never receives HERMES_API_KEY/OPENROUTER_API_KEY. Hermes never allocates builder preview ports or writes SQLite. Preview iframe origin is treated as untrusted except for nonce-checked inspector messages.

03 · UI

UI architecture

The builder UI is a Next.js App Router app on port 3001 with shadcn primitives and Luma dark tokens. Generated apps are separate Next processes framed by iframe.

Route surfaces

Route Purpose
/projects Project inventory, archive/delete, launch into builder
/projects/new Full-page project create (template + theme)
/projects/[projectId] Builder shell: chat + preview + settings
/templates Built-in + custom template inventory
/templates/new Full-page template authoring form

Primary client components

Component Role
BuilderShell useChat transport, inspect toggle, restore/edit, composer
PreviewFrame iframe src, nonce, route history, selection overlays
useBuilderUiStore Inspect mode, selected targets, screenshots, client checkpoints
Create flow forms Luma page shells for project/template create
Template inspector-provider In-preview hover/select + preferredSelector emission
Inspect model. Preferred selector is the element’s last classname (.vestaboard-title, not a shared base. The parent does not trust arbitrary iframe messages: origin + projectId + previewNonce must match.

04 · Database

Database architecture

Durable builder state is SQLite via Drizzle (lib/db/schema.ts). Workspaces are first-class filesystem trees. Git commits inside each workspace provide file-level restore points complementary to chat checkpoints.

Entity groups

Group Tables
Project core projects, conversations, runtimes, project_themes, project_settings
Chat / runs messages, runs, chat_checkpoints, session_events, hermes_session_links
Authoring registries project_templates, builder_preferences
Artifacts screenshots, project_snapshots, git_commits

Write ownership examples

createProjectAction
  → projects + conversations + runtimes
  → project_themes + project_settings
  → builder_preferences.last_opened_project_id

createCustomTemplateAction
  → project_templates (generating → ready|failed)
  → createProjectBundle for Template: edit project

POST /api/chat
  → messages(user) + runs(running)
  → messages(assistant) + runs(terminal)
  → optional hermes_session_id updates

Restore / Edit prompt
  → delete messages from clicked id forward
  → git revert workspace (disk)
  → runtime restart metadata

05 · API

API & server-action communication

AppLoop uses two complementary server entry styles: Server Actions for form mutations and navigations, and Route Handlers for streaming/binary/poll endpoints.

HTTP route handlers

Endpoint Semantics
POST /api/chat AI SDK UI message stream; bridges to Hermes; persists messages/runs
POST /api/chat/cancel Cancel active Hermes run for a project
GET /api/projects/[id]/runtime/logs Bounded runtime log polling
POST/GET .../screenshots Inspector/clipboard image attach + retrieval
GET /api/diagnostics/export Local diagnostics export

Key server actions

Action Side effects
createProjectAction DB bundle + FS copy + redirect builder
createCustomTemplateAction Template FS + Hermes once + edit project + redirect
editTemplateAction Bind template path as project, open builder
start/stop/restartRuntimeAction Preview process lifecycle on reserved port
createFileSnapshot / revertToFileSnapshot Git checkpoints for restore/edit
deleteProjectConversationMessagesFrom Timestamp/id-based chat truncation

Chat request shape (browser → AppLoop)

{
  "id": "",          // useChat id === projectId
  "messages": [ /* AI SDK UIMessage[] */ ]
}

// AppLoop extracts last user text, optional images,
// authorizes project access, then streams UI chunks back.
Why both actions and API routes? Mutations that naturally end in redirects (create project/template) stay as Server Actions. Token streaming and cancel semantics close over long-lived HTTP bodies, so they stay on Route Handlers with AI SDK stream helpers.

06 · Gateway

AI gateway communication

Hermes is reached only from the AppLoop server via lib/hermes/client.ts. Default base URL is local gateway http://127.0.0.1:8642. Auth and model routing stay off the browser.

Client methods

Method Used by
streamProjectRun Interactive builder chat (/api/chat)
runProjectOnce Template authoring (blocking until done)
cancelRun Chat cancel path
health Diagnostics / readiness

Run request payload (conceptual)

{
  projectId,
  conversationId,
  message,              // composed prompt
  workspacePath,        // path-contained
  sessionId,            // reserved or previous
  agentBundle: {
    orchestrator,
    specialists,
    skills, hooks, commands,
    mode: "project-edit"
        | "template-edit"
        | "template-authoring"
  },
  images?
}

Stream event mapping

Hermes event AppLoop handling User-visible effect
session Persist session ids on project/conversation Continuity for later turns
run Store hermesRunId on active run Enables cancel
activity UI data part + observability File/command chips under assistant message
text-delta AI SDK text stream Streaming Hermes prose
done / error Finalize run row & assistant message Completion or error banner
mode: project-edit mode: template-edit mode: template-authoring bundle from .hermes/* no browser gateway auth

07 · Flows

Core end-to-end flows

Create project (local-heavy)

UI /projects/new
  → createProjectAction
  → SQLite project bundle
  → fs.cp templates/* → PROJECTS_ROOT/slug
  → optional theme apply + git init
  → redirect /projects/:id
  // Hermes not called until first chat

Create template (gateway-heavy)

UI /templates/new
  → createCustomTemplateAction
  → validate theme CSS
  → fs.cp base → templates/
  → project_templates generating
  → hermes.runProjectOnce(template-authoring)
  → ready + openTemplateForEditing bundle
  → redirect /projects/:id

Inspect → targeted edit

Inspect toggle
  → iframe enabled
  → click .preferredSelector
  → Targets chip (last classname)
  → prompt "replace by: Josoroma"
  → createFileSnapshot (git)
  → createVisualSelectionPrompt(composed bounds)
  → POST /api/chat
  → Hermes stream edits workspace
  → assistant bubble + preview HMR/reload
Restore / Edit rewinds files via git snapshot hash, truncates UI messages and durable conversation rows from the clicked user prompt forward, restarts runtime, and (for Edit) prefills the short prompt text for resend. See docs/README-USER-FLOW-EDIT-PROJECT-OR-TEMPLATE.md.

08 · Security

Security & trust model

Hard rules

  • Never expose provider/gateway secrets to client bundles.
  • Treat browser-provided project ids, ports, routes, and Hermes session ids as untrusted inputs.
  • All filesystem mutation is path-contained (lib/security/paths.ts).
  • Runtime commands go through allow-listed helpers (lib/security/commands.ts).
  • Preview inspector requires origin + nonce match.
  • Custom themes cannot import remote CSS/URLs; token allow/ignore lists apply.

Trust zones

Trusted:     AppLoop server process
Semi-trusted: local Hermes gateway
              (workspace path is provided by AppLoop)
Untrusted:    browser form fields,
              iframe DOM messages without nonce,
              model tool arguments (still path-gated)
